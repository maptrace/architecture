Aggregator EC2

The aggregator EC2 receives requests and data from the reporter over HTTP. It authenticates the reporter and parses the request, making sure that the request is well-formed. The input data are tuples of (scur, snxt, vtyp, tseg, tend), which the aggregator EC2 writes to Kafka/Kinesis.

Aggregator Kafka

The aggregator Kafka/Kinesis collects the input tuples, buffers them and serves as the source for Flink. This process would only buffer the input for up to an hour, as Flink ought to be processing the data in close to real-time, so the buffering is only to handle issues around fault-tolerance.

Aggregator Flink

Flink uses (scur, snxt, vtyp) as a session key for a window which aggregates together (min(tend), avg(tseg), count). When count reaches npriv Flink emits the tuple (scur, snxt, vtyp, min(tend), avg(tseg), count). This is the process which enforces privacy constraints, as npriv can be set so that no single data point can be recovered from the output data. Therefore, it's a data-destroying operation, and should operate on the widest possible set of data. Data is output to a second Kafka/Kinesis instance.

DataStore Kafka

The datastore's Kafka/Kinesis instance buffers the input tuples of (scur, snxt, vtyp, min(tend), avg(tseg), count) for at least one reporting period. For example, if the reporting period is an hour, then this might buffer the data for two hours. This is to give Spark somewhere to read from, so that Spark doesn't have to stay up 100% of the time and store all this in expensive RAM. Instead, Spark is started via Lambda.

DataStore Lambda

Lambda is a source of scheduled periodic (e.g: hourly) events for Spark. Locally, we'd use cron or something for this.

DataStore Spark

Spark starts up from Lambda, drains the last period of data from Kafka, reduces it on the tile ID, and outputs ORC and FlatBuffers histogram tiles to S3.

Histogram Tiles

ORC and FlatBuffers tiles on S3, for reading by Spark.

API EC2

Receives requests from WB users and starts a Spark job to satisfy it. For bounding box requests, it looks up the segment IDs from the OSMLR tiles which are stored locally. Expands out the geometry in responses by extracting the geometry from Valhalla tiles to make GeoJSON.

API Spark

Aggregates tiles from S3 to return per-segment histograms.

Derived Data Lambda

Fires off Spark jobs when there are new Histogram Tiles or on a regular schedule.

Derived Data Spark

Reads the Histogram Tiles and strips out data which cannot be published, writing the results to a public S3 bucket.

Derived Data S3

A public bucket for derived data products.

Bulk Load S3

A private bucket into which historical trace data is uploaded for processing. This is a batch, manual process and is intended to be run rarely.

Bulk Load Spark

Loads data from the private historical trace bucket and processes it, merging it with the existing histogram trace files, and writing the results back to the histogram trace bucket.
